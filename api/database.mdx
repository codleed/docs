## Overview

LearningoPK uses **PostgreSQL 16** with **Drizzle ORM** for schema management and migrations. The database contains **22 tables** organized into 8 functional domains. All tables use `snake_case` naming. Foreign keys enforce referential integrity with appropriate cascade rules.

| Domain | Tables | Description |
|--------|--------|-------------|
| **Auth** | 4 | User accounts, sessions, OAuth providers, email verification |
| **Curriculum** | 8 | Boards, classes, subjects, chapters, content sources, cross-references, media |
| **Assessment** | 5 | Exercises, flashcards, quizzes, questions, attempts |
| **AI** | 3 | Chat sessions, messages, token usage logs |
| **Forum** | 3 | Threads, replies, votes |
| **Progress** | 1 | Per-chapter user progress tracking |
| **Admin** | 4 | Moderation flags, audit logs, notifications, settings |
| **Other** | 2 | Mock exams, institutes |

## Migration Workflow

LearningoPK uses Drizzle Kit for schema migrations against PostgreSQL.

<Tabs>
<Tab title="Generate Migration">

After modifying `backend/src/lib/db/schema.ts`, generate a migration file:

```bash
# From project root
pnpm db:generate

# Or from backend directory
pnpm drizzle-kit generate
```

This creates a SQL migration file in `backend/src/lib/db/migrations/` containing the diff between the current schema and the previous migration.

<Note>
  Always review the generated SQL before applying. Drizzle Kit generates `ALTER TABLE`, `CREATE TABLE`, and `DROP` statements based on schema changes.
</Note>

</Tab>
<Tab title="Apply Migration">

Run pending migrations against the database:

```bash
# From project root
pnpm db:migrate

# Or from backend directory
pnpm drizzle-kit migrate
```

Migrations run in order and are tracked in the `__drizzle_migrations` table. Each migration is applied exactly once.

<Warning>
  Never modify a migration file after it has been applied to a shared database. Create a new migration instead.
</Warning>

</Tab>
<Tab title="Push (Dev Only)">

For rapid development iteration, push schema changes directly without a migration file:

```bash
pnpm drizzle-kit push
```

<Warning>
  `push` is destructive and should **never** be used on production or shared databases. It drops and recreates columns/tables to match the schema.
</Warning>

</Tab>
<Tab title="Inspect Schema">

Open Drizzle Studio to browse the database visually:

```bash
pnpm drizzle-kit studio
```

This launches a local web UI at `http://localhost:4983` for inspecting tables, data, and relationships.

</Tab>
</Tabs>

### Migration Best Practices

1. **One change per migration** — Keep migrations atomic and focused
2. **Review generated SQL** — Check for unintended `DROP COLUMN` or `DROP TABLE` statements
3. **Never edit applied migrations** — Create a new migration to fix issues
4. **Seed after migrate** — Run `pnpm db:seed` after migrations that add required reference data
5. **Back up before destructive changes** — Use `pg_dump` before column renames or type changes
