## Overview

LearningoPK uses Nginx as a reverse proxy and load balancer. It handles rate limiting, gzip compression, WebSocket upgrades for AI chat, and static asset caching. The configuration lives in `infra/nginx.conf`.

## Architecture

```
                   ┌─────────────────────────────┐
  Client ────────▶ │         Nginx (port 80)      │
                   │  ┌─────────┐ ┌─────────────┐ │
                   │  │  Rate   │ │    Gzip      │ │
                   │  │ Limiter │ │  Compression │ │
                   │  └────┬────┘ └──────────────┘ │
                   └───────┼───────────────────────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         backend:3001  backend:3001  backend:3001
         (upstream)    (upstream)    (upstream)
```

## Worker Configuration

```nginx
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 2048;
    use epoll;
    multi_accept on;
}
```

<Tip>
`worker_processes auto` lets Nginx detect CPU cores automatically. `epoll` is the most efficient connection method on Linux. On Docker/WSL, these still apply.
</Tip>

## Upstream Backend

The backend runs as a single instance for development but is configured for horizontal scaling:

```nginx
upstream backend_cluster {
    least_conn;
    server backend:3001 max_fails=3 fail_timeout=30s;
    keepalive 32;
}
```

| Directive | Value | Purpose |
|-----------|-------|---------|
| `least_conn` | — | Routes to the server with fewest active connections |
| `max_fails` | 3 | Mark server as down after 3 consecutive failures |
| `fail_timeout` | 30s | Wait 30s before retrying a failed server |
| `keepalive` | 32 | Maintain 32 persistent connections to upstream |

<Note>
To scale horizontally, add more server entries to the upstream block. Nginx will distribute traffic using least-connections.
</Note>

```nginx
upstream backend_cluster {
    least_conn;
    server backend-1:3001 max_fails=3 fail_timeout=30s;
    server backend-2:3001 max_fails=3 fail_timeout=30s;
    server backend-3:3001 max_fails=3 fail_timeout=30s;
    keepalive 32;
}
```

## Rate Limiting

Two rate limiting zones are configured to protect against abuse:

```nginx
limit_req_zone $binary_remote_addr zone=ip_limit:10m rate=1000r/m;
limit_req_zone $http_x_user_id zone=user_limit:10m rate=100r/m;
```

| Zone | Key | Storage | Rate | Burst |
|------|-----|---------|------|-------|
| `ip_limit` | Client IP address | 10MB | 1000 req/min | 50 |
| `user_limit` | JWT token (X-User-ID header) | 10MB | 100 req/min | 20 |

### Application to Routes

Rate limits are applied per-location:

```nginx
location /api/ {
    limit_req zone=ip_limit burst=50 nodelay;
    limit_req zone=user_limit burst=20 nodelay;
    # ...
}
```

- **IP-based**: Catches all traffic from a single IP regardless of authentication
- **User-based**: Targets authenticated users by extracting the Bearer token from the Authorization header
- **`nodelay`**: Processes burst requests immediately instead of queuing them
- **`burst=50`**: Allows up to 50 extra requests above the rate before returning 429

<Warning>
The health check endpoint `/api/health` has rate limiting disabled. This prevents monitoring tools from being blocked.
</Warning>

### User ID Extraction

Nginx extracts the JWT token from the Authorization header for user-based rate limiting:

```nginx
location /api/ {
    set $user_id "";
    if ($http_authorization ~ "^Bearer\s+(\S+)") {
        set $user_id $1;
    }
    proxy_set_header X-User-ID $user_id;
    # ...
}
```

The extracted value is forwarded as `X-User-ID` to the backend and used as the rate limit key.

## Gzip Compression

```nginx
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 6;
gzip_min_length 1024;
gzip_types text/plain text/css text/xml application/json application/javascript
           application/xml application/xml+rss text/javascript application/x-javascript;
```

| Directive | Value | Effect |
|-----------|-------|--------|
| `gzip_comp_level` | 6 | Balance between CPU usage and compression ratio |
| `gzip_min_length` | 1024 | Don't compress responses smaller than 1KB |
| `gzip_proxied` | any | Compress responses for proxied requests |

## Buffer Optimizations

```nginx
client_body_buffer_size 128k;
client_max_body_size 10m;
proxy_buffer_size 128k;
proxy_buffers 4 256k;
proxy_busy_buffers_size 256k;
```

- **`client_max_body_size 10m`**: Limits upload size to 10MB (matches the MinIO media upload limit)
- **Proxy buffers**: Sized to handle large API responses without disk buffering

## Route Configuration

### Health Check (No Logging)

```nginx
location /api/health {
    proxy_pass http://backend_cluster;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    access_log off;
}
```

Access logging is disabled for health checks to reduce log noise.

### API Routes

```nginx
location /api/ {
    limit_req zone=ip_limit burst=50 nodelay;
    limit_req zone=user_limit burst=20 nodelay;

    proxy_pass http://backend_cluster;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_connect_timeout 60s;
    proxy_send_timeout 60s;
    proxy_read_timeout 60s;
}
```

### WebSocket (AI Chat)

The AI chat endpoint requires WebSocket upgrade headers:

```nginx
location /api/ai-chat {
    proxy_pass http://backend_cluster;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

    limit_req zone=ip_limit burst=50 nodelay;
    limit_req zone=user_limit burst=20 nodelay;
}
```

<Note>
WebSocket connections still have rate limiting applied at connection initiation. This prevents abuse while allowing long-lived streaming connections.
</Note>

### Static Assets

```nginx
location /_next/static {
    proxy_pass http://backend_cluster;
    proxy_cache_valid 60m;
    add_header Cache-Control "public, immutable";
}

location /static {
    proxy_pass http://backend_cluster;
    proxy_cache_valid 60m;
    add_header Cache-Control "public, max-age=31536000";
}
```

| Path | Cache Duration | Cache-Control |
|------|---------------|---------------|
| `/_next/static` | 60 minutes | `public, immutable` |
| `/static` | 60 minutes | `public, max-age=31536000` (1 year) |

## Timeout Configuration

```nginx
proxy_connect_timeout 60s;
proxy_send_timeout 60s;
proxy_read_timeout 60s;
```

All proxy timeouts are set to 60 seconds. This accommodates AI chat responses which can take several seconds due to Mistral API latency.

## Proxy Headers

Every proxied request receives these headers:

| Header | Source | Purpose |
|--------|--------|---------|
| `X-Real-IP` | `$remote_addr` | Original client IP |
| `X-Forwarded-For` | `$proxy_add_x_forwarded_for` | Chain of proxy IPs |
| `X-Forwarded-Proto` | `$scheme` | Original protocol (http/https) |
| `X-User-ID` | JWT token extraction | Authenticated user identifier |

## Docker Integration

The Nginx service is configured in `docker-compose.yml`:

```yaml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./infra/nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - backend
```

## Error Pages

```nginx
error_page 500 502 503 504 /50x.html;
location = /50x.html {
    root /usr/share/nginx/html;
}
```

Custom error pages can be added to `infra/html/` and mounted into the container.
